Standalone UI Preview

Widget REST editor with response bindings

This mockup shows the new control block I would add to the widget data source form so a token-producing request can bind values from its response into runtime variables for later widgets in the same Search or Refresh run.

Scope Widget-level REST data source only. No backend contract changes required.
Behavior Bindings read from successful responses and populate in-memory variables like auth_token.

Current widget data source form

Existing REST editor controls stay in place. The new block sits below the request body and above the test action.

Selected shared source or leave empty for inline REST widgets.

Response Bindings

Capture values from the JSON response and make them available as runtime variables for downstream widgets.

Must match [A-Za-z0-9_.-]+
Simple dot path, optional array index, like data.items[0].token
Bindings apply only when the widget fetch succeeds.

What changes in the UI

The editor gains one extra section instead of a separate flow.

Placement Below request headers/body, so the binding belongs to the request that produces the token.
Validation Inline helper text for variable format and path syntax, with room for field-level errors.
UX Add/remove row interaction, similar to the header editor, to keep the form consistent.
Compatibility Dashboards with no bindings look exactly the same as before.
Example chain

Token widget response: { "access_token": "tok_999" }
Downstream header: Authorization: Bearer {{auth_token}}
